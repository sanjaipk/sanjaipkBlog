Introduction:

Business Objects (BO) are instances of class that is used to represent a datum. It can also be referred to as model in newer systems. It is a basic data contract between different layers and business layer.

Database, class and collection:

Databases consists of tables which contain the following: columns (fields) - represented by properties in a class definition. rows (records) - represented by objects/instances of a class. Relationship between tables - can be enforced with inner classes or use of complex types inside a class. Set of objects is called collection and it roughly represents set of records. Most programming languages provides a lot of options to represent collections.

Business entity and Database entity:

As per enterprise architectures, database would be the first component to be designed in new application. These folks knows what are important business entities to be maintained over time. With enhancements to application, deviation between 2 entity definitions becomes greater. This requires additional processing of data (use of BO) before sending to data store.

Business entity and UI entity:

Web pages contains tons of forms that are used to capture data from end user. Due to intuitive UI designs, data from a form cannot be directly sent to data store. Converting form data to BO and letting business layer process (validation, rules, etc..) it enriches information inside a data.

Benefits:
  • Validation :
    user input acceptance in business layers can be easily achieved with operations defined in a BO class. E.g:- purchase-BO can be checked before order-BO is submitted to store.
  • Data Security :
    most of the data saving queries are exposed as functions. In most cases, business layer coder never knows about actual queries being used for DB operations. E.g:- data can be sent to cloud using secured API token over web services.
  • Transaction :
    user inputs through wizards involves building one big DB record from several tiny BO from each wizard step. Wizard can be analogized to a transaction with many steps. E.g:- profile compilation for an user (personal, financial, education details, etc.)
  • Maintain, reuse, integration :
    with respect to coding, BO improves reusability and singularity of a single business idea. It also facilitates integration with other business entities. E.g:- over time, complex user BO is created using personal BO, eductional BO, etc..
